# Кроссплатформенная консоль (рабочее название)

Современный SSH/терминальный клиент с шифрованной синхронизацией пользовательских данных через **Google Drive**. Один аккаунт, одни и те же хосты, ключи и сниппеты — на всех устройствах пользователя, без зависимости от чужого облачного бэкенда.

> Имя продукта пока не выбрано. В этом документе используется «приложение» / «консоль».

---

## Целевые платформы

- macOS
- Windows
- Ubuntu (Linux)
- iOS
- Android

Один кодбейз там, где это разумно. Платформо-специфичные API — только когда без них нельзя (биометрия, keychain ОС, фоновые задачи).

---

## Архитектурные принципы

1. **End-to-end шифрование на клиенте.** Данные шифруются на устройстве пользователя **до** загрузки в Google Drive. Drive используется как транспорт/хранилище, а не как доверенная среда. Google не имеет доступа к содержимому.
2. **Google Drive как backend.** Никакого собственного сервера, биллинга, инфраструктуры. Это сильно упрощает эксплуатацию, но накладывает ограничения на коллаборативные фичи (см. ниже).
3. **Локальный кеш + офлайн-режим.** Drive нестабилен/недоступен → приложение продолжает работать с локальной копией данных.
4. **Разблокировка vault'а биометрией** (Touch ID / Face ID / Windows Hello / Android biometrics) поверх мастер-пароля.

## Скоуп

### Must-have (MVP)

- Терминал на всех 5 платформах.
- **SSH** + **SFTP**.
- Модель данных: **Hosts**, **Groups** (с наследованием настроек), **Identities** (логин+ключ), **Keychain** для централизованного хранения ключей/паролей.
- **Known hosts**.
- **Port forwarding** (local / remote).
- **Jump hosts**.
- E2E-шифрование локального хранилища и файлов в Google Drive.
- Биометрическая разблокировка vault'а.
- Локальный кеш и офлайн-работа.

### Nice-to-have (вторая итерация)

- **Snippets** — сохранённые команды/скрипты, запуск из терминала.
- **Autocomplete** по snippets, истории, путям.
- **Session logs** с поиском и закладками.
- **Workspaces / split panes** — несколько сессий в одном окне.
- **Tags** поверх групп.
- **Command broadcasting** — одна команда сразу в N сессий.
- **FIDO2 / hardware keys**.
- AI-генерация команд из естественного языка (опционально, по API-ключу пользователя).

### Сознательно НЕ делаем (по крайней мере, пока)

- **Multiplayer / совместные сессии в реальном времени** — требует сервера-реле; несовместимо с «только Google Drive».
- **Team vaults и шаринг между пользователями** — продукт пока однопользовательский.
- **AI-агент** уровня Termius Gloria — отдельный большой проект.
- **SAML SSO, SOC 2, корпоративные фичи** — нерелевантно на этапе MVP.
- **Mosh, Telnet, Serial** — пока не приоритет.
- **Post-quantum подписи (ML-DSA)** — преждевременно.

## Открытые вопросы

Эти три развилки определяют почти всё остальное.

### 1. Технологический стэк

Реальные кандидаты на «один кодбейз для пяти платформ»:

| Стэк | Плюсы | Минусы |
|---|---|---|
| **Flutter** | Единый кодбейз для всех 5 платформ, зрелые desktop + mobile, хорошая работа с псевдотерминалом через FFI | Dart не самая популярная экосистема для системных вещей |
| **.NET MAUI** | Единый кодбейз, сильные библиотеки SSH (SSH.NET) | На Linux desktop — слабее, mobile терминал требует работы |
| **React Native + Tauri/Electron** | Огромная экосистема, много готовых компонентов | Две кодовые базы (mobile vs desktop), Electron тяжёлый |
| **Rust core + native UI** | Максимальная производительность, общая криптография и сетевой код | Сильно дольше делать, 5 разных UI-слоёв |

Предварительный фаворит: **Flutter** — балансирует «один кодбейз» и «работает на всех 5 платформах».

### 2. Схема шифрования

Стандартный, проверенный паттерн (как у Termius, 1Password и т.д.):

```
master password → KDF (Argon2id) → master key
master key → шифрует приватный ключ пользователя
random data key → шифрует фактические данные (vault)
public key пользователя → шифрует data key
```

- Симметричное шифрование: **XChaCha20-Poly1305** (или AES-256-GCM).
- KDF: **Argon2id** с агрессивными параметрами.
- Мастер-пароль никогда не покидает устройство.
- Биометрия защищает локальный кэш мастер-ключа в безопасном хранилище ОС (Keychain / Keystore / DPAPI).

### 3. Модель синхронизации в Google Drive

Drive — это файловое хранилище без транзакций и блокировок. Варианты:

- **Один зашифрованный snapshot-файл.** Просто. Конфликты при одновременной правке с двух устройств → «победил последний» или ручной merge.
- **Журнал операций (event log).** Каждое устройство дописывает свои события в свои файлы, при синхронизации все события сливаются. Корректный merge, но больше кода.
- **CRDT.** Самый правильный merge, но самый сложный.

Для MVP, скорее всего, **snapshot + блокировка через файл-маркер** хватит. Полноценный merge — позже, когда станет реальной болью.

## Источники / вдохновение

Прототипирование функциональности и UX опирается на анализ **Termius** ([termius.com](https://termius.com)) — основные идеи модели данных (hosts/groups/identities/keychain/vaults), организация snippets и логов, схема hybrid E2E-шифрования через libsodium. Принципиальное отличие нашего подхода — отказ от собственного облачного бэкенда в пользу Google Drive, что даёт независимость, но закрывает дверь к коллаборативным фичам.
